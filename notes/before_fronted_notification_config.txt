📋 Project Overview
What We're Building
A production-grade Enterprise Order Management System using Spring Boot Microservices. This is the kind of system used by companies like Amazon, Walmart, and Flipkart to manage their entire order lifecycle.

Business Problem We're Solving
text
Problem: Enterprises need to manage:
- Products across multiple categories
- Inventory across warehouses
- Customer orders end-to-end
- Payments securely
- Real-time order status tracking
- Multi-service coordination

Solution: Our OMS handles all of this with microservices!

🔐 Data Consistency & Saga Pattern
How We Maintain Consistency
text
┌─────────────────────────────────────────────────────────────────┐
│                    SAGA PATTERN FLOW                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Step 1: Create Order (PENDING)                                 │
│          ↓                                                      │
│  Step 2: Reserve Inventory (Compensatable)                     │
│          ↓                                                      │
│  Step 3: Process Payment (Compensatable)                       │
│          ↓                                                      │
│  Step 4: Confirm Order (CONFIRMED)                             │
│                                                                  │
│  ❌ If ANY step fails:                                          │
│  Step 5: Execute Compensation                                  │
│          ├── Release Inventory                                 │
│          └── Mark Order as FAILED                              │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
Consistency Strategies
Strategy	Implementation	Why Used
Eventual Consistency	Saga pattern with Kafka	Orders don't wait for payments to complete
Compensation	Release inventory on failure	Rollback changes automatically
Idempotency	Check existing payments	Prevent duplicate processing
At-least-once Delivery	Kafka consumer offsets	No message loss

🔄 Complete Order Flow (End-to-End)
Step-by-Step Flow
text
┌─────────────────────────────────────────────────────────────────────────────┐
│                    COMPLETE ORDER FLOW                                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  1. User submits order via Angular UI                                       │
│         ↓                                                                   │
│  2. API Gateway validates JWT token                                         │
│         ↓                                                                   │
│  3. Order Service receives request                                          │
│         ↓                                                                   │
│  4. Order Service calls Product Service (Feign)                            │
│     - Gets product details, price                                          │
│         ↓                                                                   │
│  5. Order Service calls Inventory Service (Feign)                          │
│     - Checks stock availability                                            │
│     - Reserves items (reduces available quantity)                          │
│         ↓                                                                   │
│  6. Order Service creates order (status: PENDING)                          │
│         ↓                                                                   │
│  7. Order Service publishes OrderCreatedEvent to Kafka                     │
│         ↓                                                                   │
│  8. Payment Service consumes OrderCreatedEvent                              │
│         ↓                                                                   │
│  9. Payment Service processes payment (simulates gateway)                  │
│     - Creates payment record (status: COMPLETED/FAILED)                   │
│         ↓                                                                   │
│  10. Payment Service publishes PaymentProcessedEvent to Kafka              │
│         ↓                                                                   │
│  11. Order Service consumes PaymentProcessedEvent                          │
│         ↓                                                                   │
│  12. Order Service updates order status:                                    │
│      - If COMPLETED → CONFIRMED                                            │
│      - If FAILED → FAILED (releases inventory)                           │
│         ↓                                                                   │
│  13. User sees order confirmation                                           │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
🔧 Resilience & Fault Tolerance
Circuit Breaker Configuration
yaml
resilience4j:
  circuitbreaker:
    instances:
      productService:
        sliding-window-size: 10        # Track last 10 calls
        failure-rate-threshold: 50     # Open circuit at 50% failures
        wait-duration-in-open-state: 5s  # Wait 5 seconds before retry
What Happens When Services Fail

┌─────────────────────────────────────────────────────────────────┐
│                    FAILURE SCENARIOS                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Scenario 1: Inventory Service Down                            │
│  → Circuit Breaker opens after 5 failures                      │
│  → Fallback returns "Stock unavailable"                        │
│  → Order fails gracefully with proper message                  │
│                                                                  │
│  Scenario 2: Payment Service Down                              │
│  → Kafka stores OrderCreatedEvent                              │
│  → When Payment Service restarts, it consumes pending events   │
│  → No order loss!                                              │
│                                                                  │
│  Scenario 3: Kafka Down                                         │
│  → Order Service still creates order                           │
│  → Payment Service retries consuming messages later            │
│  → Circuit breaker prevents cascading failures                 │

🎤 Team Lead / Teacher Explanation
"What is this project?"
"We're building an Enterprise Order Management System - the backbone of e-commerce operations. It's designed to handle the complexity of modern online retail: millions of products, real-time inventory, orders, payments, and everything that happens between 'Add to Cart' and 'Order Delivered'."

"Why Microservices?"
"We chose microservices because:

Independent scaling - Order Service can scale during sales, Product Service doesn't need to

Team autonomy - Teams can work independently on each service

Technology flexibility - Each service can use different technologies if needed

Fault isolation - Payment service failure doesn't crash the entire system"

"Why Saga Pattern?"
"Traditional distributed transactions (2PC) are slow and lock resources. Saga Pattern provides eventual consistency - orders complete faster, and if something fails, we gracefully rollback using compensation transactions."

"Why Kafka?"
"Kafka provides asynchronous, decoupled communication. If Payment Service is down during a sale, orders still get created. Events are stored in Kafka and replayed when Payment Service recovers. No lost orders!"
