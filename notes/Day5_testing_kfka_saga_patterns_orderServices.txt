Q1: Why Kafka instead of direct HTTP calls?
Answer:
"Kafka provides asynchronous, decoupled communication. If Payment Service is down, 
Order Service still works - events are stored in Kafka. When Payment Service recovers, it consumes pending events. 
With HTTP, the order would fail immediately. Kafka also handles message replay - we can reprocess events if needed."
Q2: What is a Topic?
Answer:
"A topic is like a folder or category for messages. In our system, order-created-events contains all order creation messages. 
Topics can have multiple partitions for parallel processing - we configured 3 partitions for scalability."

Q3: What happens if Kafka is down?
Answer:

"In production, Kafka runs in cluster mode (multiple servers). If one goes down, others take over. In our learning setup, 
Order Service has circuit breakers - if Kafka is unavailable, the order still succeeds (event is retried later using retry template)."

Q4: How is this better than a database?
Answer:

"A database is state (current data). Kafka is events (what happened). For microservices, events are more valuable - 
we can replay them, audit them, and let services process at their own speed. Database would create tight coupling between services."
