📖 CONCEPT 1: SERVICE DISCOVERY (Eureka)
🎓 What is Service Discovery?
In Simple Words: Think of Eureka as a telephone directory for your microservices. 
Every service tells Eureka "Hey, I'm alive at this address!" and other services look up addresses from Eureka.

📝 Why Do We Need It?
Without Eureka: We hardcode URLs like http://localhost:8082

java
// ❌ BAD - Hardcoded URL
String productUrl = "http://localhost:8082/api/products";
With Eureka: We use service names like PRODUCT-SERVICE

java
// ✅ GOOD - Service discovery
String productUrl = "http://PRODUCT-SERVICE/api/products";
💻 Where Did We Implement It?
Step 1: Eureka Server (Discovery Server)
File: discovery-server/src/main/resources/application.yml

yaml

  port: 8761  # Eureka's default port

  client:

    fetch-registry: false        # Don't fetch itself

        SpringApplication.run(DiscoveryServerApplication.class, args);

📖 CONCEPT 2: SERVICE COMMUNICATION (Feign Client)
🎓 What is Feign Client?
In Simple Words: Feign is like calling a function but that function actually lives in another service. 
It makes HTTP calls look like regular Java method calls.

📝 Without vs With Feign
Without Feign (RestTemplate - Old Way):

java
// ❌ Complex and verbose
RestTemplate restTemplate = new RestTemplate();
String url = "http://localhost:8083/api/inventory/LAPTOP-001";
ResponseEntity<InventoryResponse> response = 
    restTemplate.getForEntity(url, InventoryResponse.class);
With Feign (Our Implementation):

java
// ✅ Simple and clean! Like calling a local method
@Autowired
private InventoryClient inventoryClient;

InventoryResponse response = inventoryClient.checkStock("LAPTOP-001");

Step 3: Create Feign Client Interface
File: product-service/.../client/InventoryClient.java

java
// This interface automatically creates an HTTP client
@FeignClient(
    name = "inventory-service",  // Name in Eureka
    fallback = InventoryClientFallback.class  // Circuit breaker
)
public interface InventoryClient {
    
    // This method becomes: GET /api/inventory/{sku}
    @GetMapping("/api/inventory/{sku}")
    InventoryResponse checkStock(@PathVariable("sku") String sku);
    
    // This method becomes: POST /api/inventory/{sku}/reserve?quantity={quantity}
    @PostMapping("/api/inventory/{sku}/reserve")
    Boolean reserveStock(@PathVariable("sku") String sku, 
                         @RequestParam Integer quantity);
}
Step 4: Use Feign Client in Service
File: product-service/.../service/impl/ProductServiceImpl.java

java
@Service
@Slf4j
public class ProductServiceImpl implements ProductService {
    
    @Autowired
    private InventoryClient inventoryClient;  // Inject Feign client
    
    public ProductStockStatusResponse getProductStockStatus(String sku) {
        // This LOOKS like a local method call
        // But actually makes an HTTP call to Inventory Service!
        InventoryResponse inventory = inventoryClient.checkStock(sku);
        
        return ProductStockStatusResponse.builder()
            .availableQuantity(inventory.getAvailableQuantity())
            .inventoryAvailable(inventory.getAvailable())
            .build();
    }
}

📖 CONCEPT 3: CIRCUIT BREAKER (Resilience4j)
🎓 What is Circuit Breaker?
In Simple Words: Like an electrical circuit breaker in your house. 
If Inventory Service is failing, the circuit breaker "trips" and stops calling it, preventing cascading failures.

📝 The Problem It Solves
text
Without Circuit Breaker:
Product Service ──calls──► Inventory Service (DOWN)
                                    │
                                    ▼
                          Product Service WAITS
                          (Threads blocked)
                                    │
                                    ▼
                          All threads exhausted
                                    │
                                    ▼
                          WHOLE SYSTEM CRASHES! 💀
💻 Where Did We Implement It?
Step 1: Add Resilience4j Dependency
File: product-service/pom.xml

xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-circuitbreaker-resilience4j</artifactId>
</dependency>
Step 2: Configure Circuit Breaker
File: product-service/src/main/resources/application.yml

yaml
resilience4j:
  circuitbreaker:
    instances:
      inventoryService:  # Name of the circuit breaker
        register-health-indicator: true
        sliding-window-size: 10        # Check last 10 calls
        failure-rate-threshold: 50     # If 50% fail, OPEN circuit
        wait-duration-in-open-state: 10000ms  # Wait 10 seconds
        permitted-number-of-calls-in-half-open-state: 3
Step 3: Create Fallback Implementation
File: product-service/.../client/fallback/InventoryClientFallback.java

java
@Component
@Slf4j
public class InventoryClientFallback implements InventoryClient {
    
    @Override
    public InventoryResponse checkStock(String sku) {
        // This runs when Inventory Service is DOWN!
        log.warn("Inventory service is DOWN! Returning fallback");
        
        return InventoryResponse.builder()
                .sku(sku)
                .available(false)
                .availableQuantity(0)
                .message("Inventory service unavailable. Please try again.")
                .build();
    }
}
Step 4: Link Fallback with Feign Client
File: product-service/.../client/InventoryClient.java

java
@FeignClient(
    name = "inventory-service",
    fallback = InventoryClientFallback.class  // 👈 Link fallback
)
public interface InventoryClient {
    // ...
}
🎯 Circuit Breaker States Explained
text
     ┌─────────────────────────────────────────┐
     │                                         │
     ▼                                         │
┌─────────┐    When failures < 50%    ┌─────────┐
│ CLOSED  ├──────────────────────────►│  OPEN   │
│ (Normal)│                            │(Broken) │
└─────────┘                            └─────────┘
     ▲                                      │
     │                                      │
     │ When failures >= 50%                 │ After 10 seconds
     │                                      │
     │                                 ┌────▼─────┐
     │                                 │ HALF-OPEN│

     │                                      │
     └──────────────────────────────────────┘
        If successful, back to CLOSED
        If fails, back to OPEN
🎓 Interview Answer Ready!
Interviewer: How do you handle failures in microservices communication?

You: We implement Resilience4j Circuit Breaker pattern. When Inventory Service fails, the circuit breaker trips after 50% failure rate. 
Instead of constantly timing out, Product Service immediately returns a fallback response. This prevents thread exhaustion and cascading failures. 
The circuit breaker has three states: CLOSED (normal), OPEN (failing), and HALF-OPEN (testing recovery).
 After 10 seconds in OPEN state, it allows a few test calls to check if service recovered.

📖 CONCEPT 4: API GATEWAY (Spring Cloud Gateway)
🎓 What is API Gateway?
In Simple Words: The reception desk of your microservices. All external requests come here first, and it routes them to the right service.

📝 Responsibilities of API Gateway
text
1. Routing - Send request to correct service
2. Authentication - Validate JWT tokens
3. Load Balancing - Distribute traffic
4. Logging - Track all requests
5. Rate Limiting - Prevent abuse

spring:
  cloud:
    gateway:
      routes:
        # Route for Auth Service
        - id: auth-service

          predicates:
            - Path=/api/products/**
          filters:
            - StripPrefix=0

          uri: lb://INVENTORY-SERVICE
          predicates:
            - Path=/api/inventory/**
          filters:
            - StripPrefix=0
Step 3: Create Authentication Filter
File: api-gateway/.../filter/AuthenticationFilter.java

java
@Component
@Slf4j
public class AuthenticationFilter extends AbstractGatewayFilterFactory<Config> {
    
    @Override
    public GatewayFilter apply(Config config) {
        return (exchange, chain) -> {
            String path = exchange.getRequest().getURI().getPath();
            
            // Skip auth for public endpoints
            if (path.startsWith("/api/auth/login") || 
                path.startsWith("/api/auth/register")) {
                return chain.filter(exchange);
            }
            
            // Get and validate JWT token
            String token = extractToken(exchange.getRequest());
            if (!jwtUtil.validateToken(token)) {
                exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
                return exchange.getResponse().setComplete();
            }
            
            // Add user info to headers for downstream services
            String email = jwtUtil.extractEmail(token);
            ServerHttpRequest mutatedRequest = exchange.getRequest().mutate()
                    .header("X-Auth-User", email)
                    .build();
            
            return chain.filter(exchange.mutate().request(mutatedRequest).build());
        };
    }
}

📖 CONCEPT 5: DATABASE PER SERVICE
🎓 What is Database Per Service?
In Simple Words: Each microservice has its own private database. No service directly accesses another service's database.

📝 Why Is This Important?

🎓 Interview Answer Ready!
Interviewer: How do you manage data in microservices?

You: We follow Database per Service pattern. Each microservice has its own dedicated database. 
Product Service uses productdb, Inventory Service uses inventorydb. 
They never share databases or access each other's tables directly. 
If Product Service needs inventory data, it calls Inventory Service's API via Feign, not the database. 
This gives us loose coupling - we can change each service's database independently.

📖 CONCEPT 6: JWT AUTHENTICATION
🎓 What is JWT?
In Simple Words: JWT is like a stamped passport. Once you login, you get a token. 
You show this token for every subsequent request, and the system knows it's you without checking database every time.

📝 JWT Structure
text
eyJhbGciOiJIUzUxMiJ9.eyJzdWIiOiJ0ZXN0QGV4YW1wbGUuY29tIn0.4KQqsYqhrYNu__rzY

├─────Header─────┼─────────Payload────────┼──────────Signature──────────┤

Header: Contains algorithm info
Payload: Contains user data (email, expiration)
Signature: Ensures token wasn't tampered with
💻 Where Did We Implement It?
Step 1: Generate JWT (Auth Service)
File: auth-service/.../service/JwtService.java

java
@Service
public class JwtService {
    
    public String generateToken(String email) {
        return Jwts.builder()
                .subject(email)                    // Who is this token for?
                .issuedAt(new Date())              // When was it created?
                .expiration(new Date(System.currentTimeMillis() + 86400000)) // 24 hours
                .signWith(getSigningKey())         // Sign with secret key
                .compact();
    }
    
    public Boolean validateToken(String token) {
        try {
            Jwts.parser()
                .verifyWith(getSigningKey())
                .build()
                .parseSignedClaims(token);
            return true;
        } catch (Exception e) {
            return false;  // Token invalid or expired
        }
    }
}
Step 2: Validate JWT (API Gateway)
File: api-gateway/.../filter/AuthenticationFilter.java

// Extract token from Authorization header
String authHeader = request.getHeaders().getFirst(HttpHeaders.AUTHORIZATION);

🎓 Interview Answer Ready!
Interviewer: How does authentication work in your microservices?

You: We use stateless JWT authentication. Auth Service generates JWT tokens on login. 
All subsequent requests go through API Gateway which validates the JWT token. 
If valid, gateway forwards the request to downstream services with user info in headers. 
Services trust this because it came from gateway. This is stateless - we don't store sessions, so any instance can handle any request.

🎯 INTERVIEW PREPARATION - KEY QUESTIONS
Q1: Why Microservices instead of Monolith?
Answer: We need independent deployment, scaling, and development. 
Product team can deploy without waiting for Inventory team. 
Each service can use different technologies. If Inventory fails, Product still works.

Q2: How do you handle distributed transactions?
Answer: For our current scope, we use eventual consistency. But planning to implement Saga pattern with Kafka for order processing.

Q3: How do you debug when something fails?
Answer: Each service has unique logs. We use X-Request-Id header passed across services via Feign interceptors to trace entire request flow.

Q4: How do you handle versioning?
Answer: API Gateway routes /v1/products to Product Service v1, /v2/products to Product Service v2. Both versions can run simultaneously.

Q5: How do you secure inter-service communication?
Answer: Internal services run in private network. Gateway handles external authentication. For sensitive services, we use mutual TLS (mTLS).
