3. Feign Client (Declarative REST)
What is it? A way to call other services as if they're local methods.

Without Feign (Traditional REST):

java
// UGLY! So much boilerplate
RestTemplate restTemplate = new RestTemplate();
String url = "http://product-service/api/products/sku/" + sku;
ProductResponse product = restTemplate.getForObject(url, ProductResponse.class);
With Feign (What you built):

java
// BEAUTIFUL! Looks like a normal method call
@FeignClient(name = "product-service")
public interface ProductClient {
    @GetMapping("/api/products/sku/{sku}")
    ProductResponse getProductBySku(@PathVariable("sku") String sku);
}

// Using it:
ProductResponse product = productClient.getProductBySku("LAPTOP-002");
4. Resilience4j (Circuit Breaker)
What is it? A safety net for service failures.

Problem it solves:

text
Order Service → calls → Product Service (SLOW/DOWN)
Without Circuit Breaker: Order Service waits... and waits... TIMEOUT!
With Circuit Breaker: Quickly fails and uses fallback
Your code:

java
@FeignClient(name = "product-service", fallback = ProductClientFallback.class)
public interface ProductClient { }

// Fallback when Product Service is down
@Component
public class ProductClientFallback implements ProductClient {
    @Override
    public ProductResponse getProductBySku(String sku) {
        // Return default data instead of failing
        return ProductResponse.builder()
                .name("Product Unavailable")
                .price(BigDecimal.ZERO)
                .build();
    }
}
Circuit Breaker States:

text
CLOSED (Normal) → Calls go through
     ↓ (50% failures)
OPEN (Broken) → Calls fail immediately, don't wait
     ↓ (After 5 seconds)
HALF-OPEN (Testing) → Try one call
     ↓ (If success)
CLOSED (Back to normal)
📚 PART 3: KAFKA DEEP DIVE (Most Important!)
What is Kafka?
Simple Answer: Kafka is a message bus where services can send and receive messages asynchronously.

Analogy: Email vs Phone Call

Phone Call (Synchronous)	Email (Asynchronous/Kafka)
Must talk immediately	Can read when free
If person busy, call fails	Email stored, read later
Both must be on call	Can be processed anytime
One-to-one	One-to-many
Kafka = Email for Microservices!

Why Do MNCs Use Kafka?
Company	How They Use Kafka	Why Kafka?
Uber	Matching riders with drivers	Millions of events per second
Netflix	What you watch → recommendations	500 billion events daily
Walmart	Inventory updates across 5000 stores	Real-time stock sync
LinkedIn	Job alerts, connection suggestions	1 million+ messages/sec
Kafka vs Direct HTTP (Important!)
text
DIRECT HTTP (What you DIDN'T use for everything):
Order → Payment → Notification → Invoice
If Payment fails, Order fails! If Notification slow, Order slow!

KAFKA (What you DID use for events):
Order → [Kafka] → Payment (consumes when ready)
              → Notification (consumes when ready)
              → Invoice (consumes when ready)
If one service fails, others keep working!
Is Saga Pattern Part of Kafka?
NO! They are separate but often work together.

Concept	What it is	Do you need Kafka?
Saga Pattern	How to handle multi-service transactions	NO - can use HTTP
Kafka	How services communicate async	NO - can use HTTP

📚 PART 4: SAGA PATTERN DEEP DIVE
What is Saga Pattern?
Problem: You need to create an order that spans 3 services. If one fails, how to rollback?

Solution: Break into steps, each with a "compensation" (undo) action.

Your Saga Implementation
java
public Order processOrder(CreateOrderRequest request) {
    // STEP 1: Create order (PENDING)
    Order order = createOrder();
    
    try {
        // STEP 2: Reserve inventory (with compensation)
        reserveInventory(order);  // ← If fails, go to compensation
        
        // STEP 3: Process payment (with compensation)
        processPayment(order);
        
        // STEP 4: Confirm order
        order.setStatus(CONFIRMED);
        
    } catch (Exception e) {
        // COMPENSATION: Undo everything!
        releaseInventory(order);  // ← Put stock back
        order.setStatus(FAILED);  // ← Mark as failed
    }
    
    return order;
}
Saga vs Traditional Transaction
Traditional (2PC)	Saga (Your implementation)
Locks database rows	No locks
Slow (waits for all)	Fast (doesn't wait)
All succeed or all fail	Partial success possible
Good for banking	Good for e-commerce

Pattern 2: Saga Template (For any distributed transaction)
java
public Result doTransaction(Request request) {
    // 1. Create pending record
    Record record = createPending(request);
    
    try {
        // 2. Call first service
        callService1(record);
        
        // 3. Call second service
        callService2(record);
        
        // 4. Confirm success
        record.setStatus(SUCCESS);
        
    } catch (Exception e) {
        // 5. Compensation: Undo step 2
        undoService2(record);
        
        // 6. Compensation: Undo step 1  
        undoService1(record);
        
        record.setStatus(FAILED);
    }
    
    return record;
}

📚 PART 8: INTERVIEW PREPARATION - KEY ANSWERS
Q: How does Order Service find Product Service?
A: "Using Eureka Service Discovery. Order Service knows Product Service by name (product-service), not by IP. Eureka maintains a registry of all services and their locations. When Order Service calls productClient.getProduct(), Feign asks Eureka 'Where is product-service?' and gets the address."

Q: What happens if Inventory Service is down?
A: "Resilience4j Circuit Breaker kicks in. After 5 failures, the circuit opens. Subsequent calls fail immediately without waiting for timeout. The fallback returns default data (stock unavailable), and order creation fails gracefully with proper error message."

Q: Why did you choose Kafka over RabbitMQ?
A: "For our use case, Kafka's message persistence (messages stay for 7 days) and replay capability (can reprocess events) are valuable. If Payment Service is down for 2 hours, it can replay all missed orders when back up. Kafka also handles high throughput - Netflix processes 500 billion events daily."

Q: Explain the Saga pattern you implemented.
A: "Our Saga has 4 steps: 1) Create pending order, 2) Reserve inventory, 3) Process payment, 4) Confirm order. Each step has a compensation: if payment fails, we release inventory and mark order failed. This ensures eventual consistency without distributed locks."

Q: How do you test this system?
A: "Three levels: Unit tests for business logic, Integration tests with Testcontainers for Kafka/PostgreSQL, End-to-end via Postman. We also test failure scenarios: service down, insufficient stock, invalid products."
